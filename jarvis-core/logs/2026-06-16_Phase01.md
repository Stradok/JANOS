# Phase 1 — Infrastructure Layer (MVP) — 2026-06-16

## Summary
Built the foundational infrastructure layer for the new JANOS architecture. Created the unified LLM client, dynamic model router, hardware monitor, episodic memory system, log bootstrap, and LangGraph MVP state machine. All components import cleanly alongside the existing v2 production system.

## What Was Built

### 1. Unified LLM Client (`core/llm_client.py`)
- **BaseLLMProvider** abstract class with `chat()`, `generate()`, `list_models()`, `health()`, `pull_model()`
- **OllamaProvider** — wraps `/api/chat`, `/api/generate`, `/api/tags`, `/api/pull`, `/api/version`. Comprehensive `validate()` method that checks endpoint reachability, version, and available models. Has `api_schema()` to detect whether Ollama supports `/api/chat` or only `/api/generate`.
- **OpenRouterProvider** — optional provider for free tier models (requires user-provided API key)
- **LLMClient** singleton — auto-discovers providers, `available_model_names()` across all providers, `ensure_model()` for pull, `chat()`/`generate()` with smart routing

### 2. Dynamic Model Router (`core/model_router.py`)
- No hardcoded model names — works with whatever `ollama list` returns
- `classify_task_complexity()` — classifies user input into `routing`, `planning`, `coding`, `research`, `creative`, `chat`, `reasoning`
- `_score_model_for_task()` — scores each model against task requirements using regex patterns (small_fast/medium/large_deep/code_specialist/function_calling)
- `select_for_task()` — returns `{model, provider, task_type, reason}` with top-5 alternatives
- `ask_for_model_pull()` — logs intent to ask user before pulling models

### 3. Hardware Monitor (`core/hardware_monitor.py`)
- Background asyncio task polling at configurable interval (default 5s)
- `_parse_nvidia_smi()` — reads VRAM used/total via `nvidia-smi --query-gpu=memory.used,memory.total`
- `_parse_psutil()` — reads RAM used/total, CPU percent
- `HardwareLoad` dataclass — `is_overloaded()` (VRAM>85% or RAM>85%), `can_run_model(estimated_vram_gb)`
- `start()`/`stop()` — clean lifecycle management

### 4. Episodic Memory (`core/episodic_memory.py`)
- **EpisodicMemory** class with SQLite + ChromaDB
- Episode schema: `{id, timestamp, user_input, reasoning_steps, tool_usage, agent_selection, output, errors, corrections, outcome_score, compressed_summary, task_type, duration_ms, metadata}`
- `store_episode()` — stores episode and indexes in ChromaDB
- `search(query, k, task_type)` — RAG retrieval with vector + keyword fallback
- `get_failure_patterns(task_type)` — finds lowest-scoring episodes for pattern detection
- `compact(max_age_days, max_episodes)` — compresses old episodes, keeps top-k by score
- `_compress_episode()` — creates ~512-char summary of each episode
- `_embed()` — uses sentence-transformers if available, otherwise character-frequency fallback

### 5. Log Bootstrap (`core/log_bootstrap.py`)
- `parse_log_file()` — parses markdown log files into structured entries (headers, sections, code blocks)
- `extract_unfinished_tasks()` — regex finds in-progress/blocked tasks
- `extract_errors()` — extracts error descriptions
- `extract_key_decisions()` — extracts architectural decisions
- `LogBootstrap.bootstrap()` — scans `logs/` directory, parses all `.md` files, seeds episodic memory
- `get_reconstruction_context()` — builds LLM context string from recent unfinished tasks

### 6. LangGraph MVP State Machine (`core/state_machine.py`)
- `State` enum — all pipeline states: `INPUT → RAG_RECALL → MODEL_SELECT → TASK_DECOMPOSE → AGENT_ALLOCATE → EXECUTE → CRITIC_VALIDATE → STORE_EPISODE → RESPOND → END`
- `WorkflowContext` dataclass — mutable context passed through states
- `StateMachine` — lightweight state machine with register/run pattern
- `create_mvp_machine()` — full state registration with retry/escalate/fail edges

### 7. New API Endpoints (`main.py`)
| Endpoint | Purpose |
|----------|---------|
| `GET /api/status` | Phase 1 system status (hardware, memory, LLM) |
| `POST /api/chat` | MVP chat pipeline (RAG recall → model select → respond → store episode) |
| `GET /api/memory/search` | Semantic search over episodic memory |
| `GET /api/memory/stats` | Episodic memory statistics |
| `GET /api/memory/failures` | Failure patterns by task type |
| `GET /api/models` | List available Ollama models |
| `POST /api/models/validate` | Validate Ollama endpoint |

## Architecture Notes
- v2 production system is untouched and still the default orchestrator
- Phase 1 components are wired into FastAPI's `app.state` with clean lifecycle
- All new endpoints are at `/api/*` to avoid conflicts with existing v1/v2 routes
- Episodic memory uses its own ChromaDB collection (`chroma_episodic/`) separate from the legacy v2 ChromaDB
